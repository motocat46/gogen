# pkg/generator — 设计文档

> 本文档记录实现细节。slice/map 不暴露整体 getter 的决策背景见 [DECISIONS.md D-003](../../DECISIONS.md#d-003-slicemap-不暴露整体-getter)；plain 模式的设计理由见 [DECISIONS.md D-006](../../DECISIONS.md#d-006-plain-模式为什么需要哪些方法被裁剪)；override 的设计理由见 [DECISIONS.md D-009](../../DECISIONS.md#d-009-override-tag强制覆盖提升方法检查)。

## 设计演进

### v1：单函数 switch
最初生成逻辑全在一个大型 switch 函数内，每种类型对应一个 case 分支。问题：
- 添加新类型需修改中心函数，违反开闭原则
- 单一模板文件超过 400 行，难以定位和调试特定类型的生成逻辑

### v2：Registry 模式（当前）
将每种类型的生成逻辑拆分到独立的生成器文件，由 Registry 管理映射关系。
- 新增类型：实现接口 + 注册，无需改动已有代码
- 每个文件只包含一种类型的模板和逻辑，职责单一

## 关键设计决策

### 1. 为什么用 text/template 而非字符串拼接

**理由**：
- 模板直观展示生成代码的结构，可读性远优于 `fmt.Sprintf`
- 条件渲染（`{{ if .GetLen }}`）映射到生成决策，逻辑清晰
- 未来修改方法签名、注释格式时，只需改模板字符串，不碰控制逻辑

**代价**：模板执行错误只能在运行时发现。解决方式：`template.Must` 在包初始化时 panic，及早暴露模板语法错误。

### 2. 生成器不持有状态

所有生成器（`BasicGenerator`、`SliceGenerator` 等）都是无字段的空结构体，`Generate` 方法是纯函数（输入相同 → 输出相同）。这是幂等性的基础：同一 `StructDef` 多次生成结果完全一致。

### 3. plain 模式的过滤粒度

plain 模式（`gogen:"plain"` 或结构体级 `// gogen:plain` 注释）的过滤在生成器内部完成，而非在分析层过滤字段：
- 分析层只管解析结构，不做生成决策
- 生成器负责根据 `f.Config.Plain` 决定哪些方法跳过
- 每个生成器独立实现 plain 规则，easy to customize

plain 模式各类型的裁剪规则详见 [DECISIONS.md D-006](../../DECISIONS.md)。

### 4. 方法名冲突的三层检查

生成器通过 `resolveCanGen(s, f)` 获取检查函数，每次生成一个方法名前调用：

```
canGen("GetXxx")  →  StructDef.CanGenerateMethod("GetXxx")
```

三层检查顺序（短路）：
1. 与字段名相同（Go 编译器禁止）
2. 手写文件已有同名方法（避免重复声明）
3. 嵌入提升方法同名（会覆盖提升语义，破坏接口实现）

`override` 标签跳过第三层，由 `CanGenerateMethodOverride` 处理。详见 DECISIONS.md D-009。

### 5. 不暴露 slice/map 的整体 getter

SliceGenerator 和 MapGenerator 不生成返回整个集合引用的方法（如 `GetTags() []string`），只提供细粒度操作：
- 防止调用方直接 `append` 绕过封装
- 强制通过受控接口修改集合
- 数组（值类型）例外：`GetXxx() [N]T` 返回的是拷贝，不存在引用泄露问题

详见 DECISIONS.md D-003。

### 6. 文件头模板不含时间戳

```go
var fileHeaderTmpl = `// Code generated by gogen; DO NOT EDIT.`
```

不含时间戳，确保相同输入产生相同字节，是增量跳过和幂等性的前提。设计理由详见 [DECISIONS.md D-010](../../DECISIONS.md#d-010-增量生成内容对比跳过写入)。

## 模板变量说明

所有模板通过 `map[string]any` 传入数据，关键字段：

| 变量 | 类型 | 含义 |
|------|------|------|
| `ReceiverType` | string | 接收者类型，如 `"Cache[K, V]"` 或 `"User"` |
| `MethodName` | string | 与字段名相同，如 `"Tags"` → 方法名 `"GetTagsAt"` |
| `FieldName` | string | 字段名，与 MethodName 当前完全相同 |
| `ElemType` | string | slice/array 的元素类型字符串 |
| `Doc` | string | 已格式化为 `// xxx` 的注释字符串 |
| `Any` | bool | 是否有任何方法需要生成（控制文档注释是否输出） |
| `GetAt` / `SetAt` 等 | bool | 对应方法是否需要生成（由 canGen + 权限检查决定） |

## 扩展：新增自定义类型生成器

以添加对 `chan` 类型的只读生成器为例（目前被跳过）：

```go
type ChanGenerator struct{}

func (g *ChanGenerator) Generate(s *model.StructDef, f *model.FieldDef) ([]byte, error) {
    canGen := resolveCanGen(s, f)
    if !canGen("Recv"+f.Name) {
        return nil, nil
    }
    // 使用 text/template 或字符串拼接生成代码
    return []byte(fmt.Sprintf(`
// Recv%s 从 channel %s 接收一个值
func (this *%s) Recv%s() (%s, bool) {
    v, ok := <-this.%s
    return v, ok
}
`, f.Name, f.Name, s.ReceiverType(), f.Name, elemType, f.Name)), nil
}

// 注册（在 main.go 或测试中）
reg := generator.NewRegistry()
reg.Register(model.KindUnsupported, &ChanGenerator{}) // 覆盖默认的"跳过"行为
```

## 已知陷阱

### formatDoc 的多行注释处理

`ast.CommentGroup.Text()` 剥离了 `//` 前缀，返回纯文本。若直接用 `"// {{ .Doc }}"` 渲染，多行注释第二行起缺少 `//`，导致生成代码语法错误。`formatDoc` 函数为每行重新添加 `//` 前缀，需要在所有传入 `Doc` 的地方调用。

### 生成器返回 nil vs 空字节

生成器返回 `nil` 时，`GenerateStruct` 不会将其计入有效输出（通过 `bytes.TrimSpace` 判空）。若某字段所有方法均被跳过，整个方法体可能为空，此时 `GenerateStruct` 返回 `nil, nil`，通知 writer 层不必创建文件。
